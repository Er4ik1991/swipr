# Swipr — бриф для Claude Code: эволюция к HH-уровню данных при сохранении свайпа и простоты

**Контекст.** У нас есть рабочий кликабельный прототип Swipr (Tinder для работы) в одном файле:
двусторонние свайпы (соискатель листает вакансии, компания — кандидатов), `DATA.seeker` /
`DATA.company`, очередь `queue`, массив `matches`, навигация `go(id)`, нижний таббар
`Лента / Отклики / Профиль`, модалка мэтча. Нужно развить продукт по мотивам HH.ru, **не сломав
две его главные ценности: свайп и простоту.**

---

## 0. Главный принцип (не нарушать)

> **HH мы используем как модель ДАННЫХ, а не как модель ИНТЕРФЕЙСА.**
> HH — информационно-плотный джобборд (длинные фильтры, списки, много текста). Swipr — обратное:
> один объект = один экран, действие = жест. Любая фича из HH переносится только если её можно
> подать через свайп, одну карточку или один тред чата. Если фича требует таблицы/портянки полей —
> она прячется за «Подробнее» или не делается вовсе.

Чек-лист перед добавлением любого экрана:
1. Можно ли это сделать жестом или одним тапом? Если нет — упростить.
2. Помещается ли основное на один экран без скролла-простыни? Если нет — вынести в раскрывающуюся панель.
3. Не дублирует ли это существующий флоу? Если да — не плодить.

## 1. Новая нижняя навигация (ядро задачи)

Сейчас: `Лента / Отклики / Профиль`.
Стало: **`Лента / Чаты / Профиль`** — три вкладки, не больше.

«Отклики» переименовываем и превращаем в полноценные **«Чаты»** (как в Tinder): это список тех,
с кем образовался мэтч, плюс те, кто пригласил на первое общение и на собеседование.

Структура вкладки «Чаты» (одна прокручиваемая страница, сверху вниз):
- **Лента новых мэтчей** — горизонтальный ряд аватарок-кружков сверху (как в Tinder «New Matches»).
  Это свежие взаимные свайпы, по которым ещё не было переписки.
- **Список диалогов** — вертикальный список тредов: аватар, имя/должность (или название
  компании/вакансии), последнее сообщение, время, бейдж непрочитанных.
- На каждом треде — **статус-чип** текущего состояния (см. стейт-машину в разделе 4):
  `Общаемся` / `Приглашение на собеседование` / `Оффер` / `Отказ`.
- Бейдж счётчика непрочитанных на иконке таба (у нас уже есть `navCount` — переиспользовать).

Тап по мэтчу/треду → **экран чата** (новый, раздел 3).

## 2. Модель данных (адаптация сущностей HH)

Завести нормальные объекты в стейте (пока можно в памяти/localStorage прототипа, но со структурой,
готовой к реальному бэкенду). Имена полей сознательно близки к HH — это упростит будущую интеграцию.

```js
// Профиль (две роли в одном аккаунте, переключение)
Profile = {
  id, role: 'seeker' | 'employer',
  // seeker:
  name, photo, title, experienceYears, salaryFrom, city, format,
  about, skills: [skillId], documents: [{title, org, year, fileUrl, verified}],
  hiddenFields: ['names_and_photo' | 'phones'],   // ОТЛОЖЕНО: в прототипе не используем, всё открыто
  completeness: 0..100,                            // индикатор заполненности
  verified: bool
  // employer:
  // company: {logo, name, industry, size, about, verified}
  // vacancies: [Vacancy]
}

Vacancy = {
  id, title, salaryFrom, salaryTo, format, city,
  description, requirements: [skillId], level, status: 'active'|'paused'|'closed',
  hasTest: bool, moderation: 'pending'|'approved'|'rejected', boosted: bool
}

// Справочники (единые словари вместо свободного ввода — как areas/skills у HH)
Dictionary = { skills: [{id, name}], areas: [{id, name}], formats: [...], levels: [...] }

// КЛЮЧЕВАЯ СУЩНОСТЬ — «отклик/приглашение» = тред чата (по образцу HH negotiation)
Negotiation = {
  id,
  initiator: 'seeker' | 'employer',  // отклик (соискатель) или приглашение (работодатель)
  seekerId, employerId, vacancyId, resumeId,
  state: NegotiationState,           // см. раздел 4
  unread: int,
  lastMessageAt,
  hidden: bool,                      // «скрыть тред», как hide у HH
  messages: [Message]
}

Message = {
  id, authorId, text, attachments: [],
  createdAt, readAt,
  // системное сообщение, меняющее состояние треда (как у HH: смена state = сообщение)
  systemState: NegotiationState | null,   // null = обычное сообщение, не меняет state
  interview: {
    datetime,
    format: 'video' | 'office' | 'phone',
    place,                 // адрес офиса (если format='office')
    link,                  // ссылка на видеозвонок (если format='video')
    status: 'proposed' | 'accepted' | 'declined' | 'rescheduled'
  } | null  // прикреплённое приглашение на собеседование
}
```

**Решение по приватности (зафиксировано для прототипа):** анкета соискателя открыта всегда —
имя/фото/телефон видны до мэтча. Поле `hiddenFields` в модели оставляем как задел на будущее, но
в логике первой версии не используем. (В реальном продукте — крючок монетизации, включим позже.)

## 3. Экран чата (новый)

Минималистичный мессенджер, не перегружать.
- Шапка: аватар + имя/должность (или вакансия/компания), статус-чип состояния, кнопка «назад».
- Лента сообщений: пузыри слева/справа, статусы прочтения, время.
- **Системные сообщения по центру** при смене состояния: «Вы пригласили на собеседование»,
  «Приглашение принято» и т.п. (это `Message.systemState != null`).
- **Карточка-приглашение на собеседование** прямо в треде — полноценная:
  - выбор формата: `Видеозвонок` / `Офис` / `Телефон`;
  - дата и время;
  - для `Видеозвонок` — поле ссылки (можно вставить свою или сгенерировать заглушку-ссылку),
    в треде ссылка показывается кнопкой `Присоединиться`;
  - для `Офис` — адрес/место;
  - кнопки получателя: `Принять` / `Предложить другое время` / `Отклонить`.
  Это главный «деловой» элемент чата.
- Поле ввода: текст, вложение (документ/файл), 2–3 шаблона быстрых ответов
  («Расскажите подробнее», «Когда удобно созвониться?»).
- Действия в меню треда: `Пригласить на собеседование`, `Отклонить`, `Скрыть тред`, `Пожаловаться`.
  (Для employer-роли — ещё `Передать коллеге` как заглушка под будущее.)

## 4. Стейт-машина треда (сердце фичи — взять у HH negotiation)

Состояния (`NegotiationState`), общие для обеих сторон:

```
matched            → взаимный свайп, тред создан, переписки ещё нет (это «Новый мэтч»)
talking            → пошла переписка («Общаемся»)
interview_invited  → отправлено приглашение на собеседование (карточка-приглашение в треде)
interview_set      → приглашение принято, время согласовано
offer              → сделан оффер
rejected           → отказ любой из сторон
hidden             → тред скрыт пользователем (не удалён)
```

Правила переходов (повторяем логику HH: смену состояния всегда сопровождает системное сообщение;
обычные сообщения состояние не меняют):
- `matched → talking`: при первом текстовом сообщении любой стороны.
- `talking → interview_invited`: сторона жмёт «Пригласить на собеседование», в тред падает
  карточка-приглашение (`Message.interview`).
- `interview_invited → interview_set`: получатель нажал «Принять».
- `interview_invited → talking`: «Предложить другое время» (возврат к переписке с контр-предложением).
- любое → `rejected`: «Отклонить» (с опциональной причиной).
- любое → `hidden`: «Скрыть тред».
- `interview_set → offer`: работодатель присылает оффер.

Группировка во вкладке «Чаты»:
- ряд «Новые мэтчи» = состояние `matched`;
- список диалогов = `talking / interview_invited / interview_set / offer`, отсортированы по `lastMessageAt`;
- `rejected` и `hidden` уходят вниз / в архив, не мозолят глаза.

Статус-чип на треде = человекочитаемое имя состояния (см. раздел 1).

## 5. Что ещё взять у HH — но строго дозированно (фазы 2–3)

Только после того, как чаты работают. Каждая фича — минимальной кнопкой/панелью, без портянок.

- **Единые справочники навыков/городов** вместо свободного ввода в анкетах (autocomplete-чипы).
  Это резко повышает качество матчинга и убирает мусорный ввод. Берём прямо модель HH `skills/areas`.
- **Фильтры ленты** — но за иконкой ⚙, в виде компактной шторки (должность, вилка, формат, город,
  опыт), а не постоянной боковой панели как у HH. Часть фильтров — платные (Plus).
- **Дневные лимиты свайпов/приглашений** для Free-тарифа (у HH это лимит просмотров резюме) — это и
  бизнес-логика, и защита от спама. Заложить счётчик в стейт.
- **Верификация**: бейдж «подтверждено» на документах соискателя и «проверенный работодатель» —
  один значок на карточке, не отдельный экран.
- **Модерация** вакансий/анкет: статус `pending/approved/rejected`, не показывать в ленте до `approved`.
- **Сохранённые поиски / избранное** (у HH есть favorites + saved searches) — опционально, тихая фича.
- **Токены дизайна**: вынести цвета/отступы/радиусы в CSS-переменные (по образцу Magritte:
  группы Background/Typography/Controls/Notifications, шаг отступов 4px), подготовить тёмную тему.

Чего **НЕ** делать (анти-раздувание):
- никаких длинных списков вакансий «как на HH» — лента остаётся свайповой;
- не выносить фильтры в постоянную панель;
- не добавлять 4-ю, 5-ю вкладку — навигация остаётся из трёх пунктов;
- не превращать анкету в многостраничную форму HH — оставить короткую с индикатором заполненности.

## 6. План работ для Claude Code (по фазам, с критериями приёмки)

### Фаза 1 — Вкладка «Чаты» и стейт-машина (приоритет)
1. Переименовать таб `Отклики → Чаты`, обновить иконку и оба места таббара (`swipe`-экран и сам экран чатов).
2. Завести объект `Negotiation` и список `negotiations` в стейте; при мэтче (взаимный свайп)
   создавать тред в состоянии `matched`.
3. Сверстать вкладку «Чаты»: ряд «Новые мэтчи» (кружки) + список диалогов со статус-чипами и бейджами непрочитанных.
4. Реализовать переходы состояний из раздела 4, каждый переход = системное сообщение в треде.
- **Приёмка:** взаимный свайп создаёт мэтч → он появляется в «Новых мэтчах» → первое сообщение
  переводит в «Общаемся» и двигает тред в список диалогов; счётчик непрочитанных корректен.

### Фаза 2 — Экран чата и приглашение на собеседование
5. Экран чата (раздел 3): лента сообщений, ввод, шаблоны, вложения-заглушки.
6. Карточка-приглашение на собеседование: создание (формат `Видеозвонок/Офис/Телефон` + дата/время +
   ссылка для видео / адрес для офиса) и реакции `Принять / Другое время / Отклонить` со сменой состояния.
   Для видео — в треде кнопка `Присоединиться` по ссылке.
7. Меню треда: `Скрыть`, `Отклонить`, `Пожаловаться`.
- **Приёмка:** можно пройти весь путь `matched → talking → interview_invited → interview_set → offer`
  и обратные ветки; системные сообщения и статус-чип отражают состояние.

### Фаза 3 — Данные и качество матчинга
8. Справочники навыков/городов в анкетах (autocomplete-чипы), привязка к матчингу.
9. Компактные фильтры ленты за иконкой; дневные лимиты для Free.
10. Бейджи верификации (документы соискателя, проверенный работодатель); статусы модерации
    вакансий/анкет (`pending/approved/rejected`, в ленте только `approved`).
    (Приватность анкеты до мэтча — НЕ в этой версии, см. раздел 2.)
11. Дизайн-токены (CSS-переменные) + заготовка тёмной темы.
- **Приёмка:** ввод навыков только из справочника; лента уважает фильтры и лимиты; бейджи
  верификации и статусы модерации работают; цвета/отступы идут через токены.

## 7. Привязка к существующему коду

- Переиспользовать `go(id)` для навигации на новый экран чата (`go('chat')` с передачей id треда).
- `matches[]` → мигрировать в `negotiations[]` (мэтч становится тредом с состоянием `matched`).
- Модалку мэтча оставить, но её кнопка «Написать сообщение» теперь открывает экран чата
  (создаёт/открывает тред), а не просто закрывает модалку.
- `navCount` / `navCount2` → завязать на сумму `unread` по тредам.
- Бейджи и счётчики держать в одном месте (один `render`-проход по `negotiations`).
